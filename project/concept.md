Концепция выпускной работы по курсу "Дизайн сетей ЦОД"

Проблема:
- наша компания арендует стойки в PoD-ах рзличных датацентров
- хостер уже предоставляет сетевую связность внутри PoD-а
- хостер имеет стандартные EVPN VXLAN IP-фабрики, построенные по топологии CLOS
- в одной стойке есть 2 коммутатора уровня leaf
- каждый сервер PoDключается двумя портами к этим коммутаторам, по одному порту в каждый leaf-коммутатор
- мы не можем управлять этой фабрикой, поэтому мы не можем строить свои Overlay-сети, используя эти фабрики
- необходимо разработать решение, которое позволит строить собственные Overlay-сети в текущих условиях

Задача:
необходимо провести эксперимент по созданию собственной Overlay-сети, построенной на базе технологий SDN, предоставленной системой виртуализации Proxmox, поверх Overlay-сетей PoD-ов хостеров и собственного inter-PoD и inter-DC транспортных линков

Общая схема ![Схема сети](scheme.png)

- в лабораторной работе представлена эмуляция трёх PoD-ов, расположенных в двух разных городах
- каждый PoD состоит из CLOS IP-фабрики построенной по технологии EVPN XLAN на обордовании Arista
- два PoD-а (номер 1 и номер 2) принадлежать одному хостеру и находятся в одном городе в одном Датацентре
- третий PoD - это другой хостер и другой Датацентр в другом городе
- Так как Хостер не предоставляет связность между PoD-ами, то PoD-ы 1 и 2 имеют BGW (border gateway) Leaf коммутаторы, которые PoDключены к нашим inter-PoD коммутаторам
- для транпортировки трафика в другой датацентр в другом городе inter-PoD коммутаторы PoDключены к нашим inter-DC роутерам
- арендовано два канала для inter-DC трафика
- так как в другом городе есть только один PoD, то inter-PoD коммутаторы не нужны, поэтому установлены только наши inter-DC роутеры, в которые приходят два арендованных канала связи из города 1
- в каждом PoD-е в каждой арендованной стойке стоит по одному серверу Proxmox (который PoDключен двумя сетевыми картами к стоечным leaf-коммутаторам)
- также, на весь PoD выделены два сервера, они называются dcX-podX-evpn-rt-X
- назначение этих серверов - быть route-reflectors для кластеров Proxmox, а также быть шлюзами (BGW) для внутренних SDN-networks, которые существуют внутри SDN Proxmox, чтобы эти сети могли общаться с нашими сетями, которые существуют в нашем ядре и в других локациях.
- IP-фабрики хостеров могут PoDнимать BGP-соединение с серверами dcX-podX-evpn-rt-X, для маршртизаци нашего трафика между нашим ядром и серверами внутри PoD-а
- BGW-leaf switches хостеров PoDнимают BGP-соединения с нашими inter-PoD коммутаторами или inter-DC роутерами для маршрутизации агрегированных префиксов между фабрикой и нашим ядром
- на каждый PoD для наших серверов выделяется префикс /16 из диапазона 10.0.0.0/8
- внутри фабрики создан VRF для нашего Тенанта
- PoDключенные к leaf-коммутаторам сетевые карты серверов объединяются в bonding.
- со стороны leaf-коммутаторов хостера эти порты объедииняются в Multi-houming
- BGP ASNs:
  - DC-X: our core  - 65101
  - DC-X: our inter-PoD switches - 65102
  - DC-X: hoster IP-fabric PoD1 - 64501 (VRF TENANT-X)
  - DC-X: our EVPN-routers PoD1 - 65111
  - DC-X: hoster IP-fabric PoD2 - 64502 (VRF TENANT-Y)
  - DC-X: our EVPN-routers PoD2 - 65112

  - DC-Y: our core  - 65201
  - DC-Y: hoster IP-fabric PoD3 - 65503
  - DC-y: our EVPN-routers PoD3 - 65113


- Inter PoD switches - Huawei CE12000 - (interpos-rt-1, interpod-rt-2)
- Inter DC routers - Huawei NE40 (city1-core-rt1, city1-core-rt2, city2-core-rt1, city2-core-rt2)
- podX-evpn-rt-X - Linux 

- Названия серверов Proxmox: dcX-pod1-cl1-prox-1
  dcX - datacenter X
  pod1 - PoD number
  cl1 - Cluster number
  prox-1 - server number in cluster


Hoster Underlay никак не связан между PoD-ами, работает ТОЛЬКО в пределах одного PoD (одной фабрики)
